# Zoomed-out density pyramid

## Problem

The web map computes zones and anchors **per viewport**. Zoomed in, that works.
Zoomed out, the viewport covers so much terrain that the raw data is either
refused (`/anchors` and, for the chunked layout, `/zones` 413 past
`MAX_VIEW_CHUNKS = 64` overlapping chunks) or would be a useless wall of tiny
polygons. Today the user just sees "zoom in to see zones/anchors" ‚Äî a dead end.

We want the zoomed-out view to answer **"where should I zoom in?"** by showing
**where the candidate-highline hotspots are**, drawn as **grid cells shaded by
count**.

## Approach

**Static, precomputed density pyramid** ‚Äî one cell layer per zoom level,
generated by a new offline Python step from the already-precomputed pairs, and
served to the browser as GeoJSON cells. The raw per-pair data being too large to
send at low zoom is exactly the problem, so aggregation happens offline and the
result (a few hundred cells per zoom) is tiny.

Rejected alternatives:
- **Live per-request aggregation** ‚Äî reading every overlapping chunk's pairs on
  the fly is the read `MAX_VIEW_CHUNKS` already refuses at low zoom; slow exactly
  where we need it.
- **Client-side clustering** ‚Äî requires sending the raw points first, which is
  the thing we can't send at low zoom.

## Cell scheme: web-mercator tile grid

Cells are slippy-map tiles `(z, xtile, ytile)` ‚Äî the same grid Leaflet/OSM
already use. Consequences:
- Each Leaflet zoom maps 1:1 to a precomputed density layer.
- Cells always tile the viewport cleanly; no seams against the basemap.
- Cell ‚Üí polygon is just the tile's lon/lat bounds (standard slippy-map math).
- At z‚âà12 a tile is ~10 km (matches `CHUNK_M`), getting coarser as you zoom out.

Precomputed zoom range: **z6‚Äìz12** (inclusive). The detailed per-viewport zone
view takes over above the top of this range (see `DENSITY_MAX_ZOOM`). Display
zooms below z6 clamp to the z6 layer.

## Components

### 1. Density builder ‚Äî `highliner/services/density.py`

`build_density(region_dir: Path, zoom_levels: Iterable[int], report=None) -> int`

- Reads `grid.json` to enumerate the region's chunks.
- Streams every `pairs/q_{cx}_{cy}.parquet` once via
  `repositories.candidates.load_candidates`.
- For each candidate pair, computes a **representative point = midpoint of the
  two anchors** (`(a.x+b.x)/2, (a.y+b.y)/2` in UTM), converted to lon/lat via
  `core.geo.to_lonlat`. Midpoint puts the density where the gap actually is.
- For **each zoom level**, computes the containing tile `(xtile, ytile)` and
  accumulates, per `(z, xtile, ytile)` cell:
  - `n` ‚Äî count of pairs (the color metric),
  - `max_exp` ‚Äî max `exposure` across the cell's pairs (carried for tooltip).
- Writes one file per zoom: `data/<region>/density/z{z}.json`, each a compact
  list of non-empty cells: `[{"x": ..., "y": ..., "n": ..., "max_exp": ...}, ...]`.
- Returns the number of cells written (for the CLI summary line).
- `report(done, total)` callback drives progress, matching the elapsed/ETA
  reporting `precompute-catalonia` uses.

A `zoom_levels` default of `range(6, 13)` lives in `core/config.py` as
`DENSITY_ZOOM_LEVELS`, alongside `DENSITY_MAX_ZOOM` (the frontend threshold).

Filtering: the stored pairs already reflect the default pairing params, so the
density is a fixed scouting layer at defaults ‚Äî not slider-reactive. This is
called out to the user in the UI framing (a "where to look" hint; exact
slider-filtered results appear on zoom-in).

### 2. CLI subcommand + `just` recipe

Mirrors the `precompute-catalonia` pattern:
- `cli.py`: `_cmd_precompute_density(args)` reads `--region` (default
  `catalonia`) and `--data-dir`, calls `build_density`, prints a summary line.
  Registered as the `precompute-density` subparser.
- `justfile`:
  ```
  # Build the zoomed-out density pyramid from precomputed pairs.
  precompute-density *args:
      uv run highliner precompute-density {{args}}
  ```

### 3. Endpoint ‚Äî `highliner/router/density.py`

`GET /density?region=&z=&bbox_lonlat=` (also accepts UTM `bbox` for symmetry):
- Loads `data/<region>/density/z{z}.json` (clamping `z` into the precomputed
  range; 404 if the region has no density dir).
- Builds a GeoJSON `FeatureCollection` of cell polygons (tile lon/lat bounds),
  clipped to the requested bbox, each feature carrying `n_pairs` and
  `max_exposure`.
- Registered in `app.py` next to the other routers. Tile‚Üîlon/lat helpers live
  in `core/geo.py` (or a small `core/tiles.py`) so the builder and router share
  them.

### 4. Frontend ‚Äî `web/app.js`

- New `DENSITY_MAX_ZOOM` constant (top of the range, e.g. 12) and a
  `densityLayer` (`L.geoJSON`) with a choropleth style keyed on `n_pairs`
  (reuse the yellow‚Üíred ramp; scale by count).
- `refresh()` branches on `map.getZoom()`:
  - **‚â§ `DENSITY_MAX_ZOOM`**: hide the zone `layer`, fetch
    `/density?region=&z=<zoom>&bbox_lonlat=<bbox>`, render cells. Tooltip:
    "`n` candidate lines ¬∑ up to `max_exp` m". Status: "N hotspot cells".
  - **> `DENSITY_MAX_ZOOM`**: clear `densityLayer`, run today's zone flow
    unchanged.
- Clicking a cell (or zooming past the threshold) lands in the detailed zone
  view ‚Äî no special handling needed beyond the existing zoom-driven `refresh()`.

## Testing

- `services/density.py`: unit test that a handful of synthetic candidate pairs
  at known coordinates land in the expected tiles and produce the expected
  `n`/`max_exp` per zoom (including two pairs sharing a cell, and a pair whose
  midpoint sits away from either anchor).
- Tile math: round-trip / known-value test for lon/lat ‚Üî tile at a couple of
  zooms and a Catalonia coordinate.
- `router/density.py`: test that `/density` returns clipped cell polygons with
  the right properties, clamps out-of-range `z`, and 404s a region with no
  density dir.

## Out of scope

- Slider-reactive density (would need per-cell exposure/length histograms).
- Anchor-count density mode (pairs are the chosen signal; `max_exp` is carried
  but not yet a selectable color metric).
- Regenerating density automatically after web-triggered `/analyze`; the builder
  is a manual batch step like `precompute-catalonia`.
